# AI 海龟汤游戏 PRD v1.0

> 核心命题：让玩家在卡死时感受到张力，而不是放弃。

---

## 1. 产品概述与核心设计原则

AI海龟汤游戏是一款由AI担任主持人的推理解谜网页游戏。玩家阅读「汤面」（一段简短的悬疑情境），然后通过向AI提问，逐步推理出隐藏的「汤底」真相。

**核心体验目标：** 让玩家经历完整的情感曲线——

> 「迷惑 → 摸索 → 卡顿 → 张力蓄积 → 突破 → 恍然大悟」

这条曲线不会自然发生。它需要被设计出来，核心工具是AI主持人的行为机制。

### 不可妥协的设计原则

| 原则 | 含义 | 优先级 |
|------|------|--------|
| 张力曲线优先 | 游戏体验必须有起伏，不能让玩家顺滑地一路问下去 | P0 |
| 卡死不等于失败 | 玩家卡住是正常状态，AI必须把卡死转化为「快要突破了」的感觉 | P0 |
| 不能让玩家主动放弃 | 玩家求助或卡死超过阈值，AI必须介入，宁可破格也不能流失玩家 | P0 |
| 沉浸感不能被打破 | AI的每一句话都必须有氛围感，不能像查数据库 | P1 |
| 难度期望必须匹配 | 玩家选了什么难度，就得到对应的挑战强度，不能言不符实 | P1 |

## 2. 目标用户

| 用户类型 | 核心需求 | 最怕什么 | 难度偏好 |
|---------|---------|---------|---------|
| 推理爱好者 | 题目有深度，推理过程有层次感 | 10分钟秒杀，没有挑战感 | ⭐⭐⭐⭐~⭐⭐⭐⭐⭐ |
| 轻度休闲玩家 | 30分钟内能推进游戏，有成就感 | 卡死超过10分钟，直接离开 | ⭐⭐~⭐⭐⭐ |
| AI体验者 | AI有个性，交互有惊喜感 | AI像机器人，冷冰冰回答 | ⭐⭐⭐（任意） |

## 3. 核心机制：AI主持人行为设计 ⭐

> 这是整份PRD最重要的章节。游戏体验的好坏，90%取决于AI主持人的行为是否符合以下设计。

### 3.1 卡死检测算法

系统需要在每轮对话后实时判断玩家是否进入卡死状态：

| 判断维度 | 触发条件 | 类型 |
|---------|---------|------|
| 无效问题连发 | 连续5个问题AI回答均为「否」或「无关」 | 主要触发条件 |
| 重复提问 | 同一方向的问题语义相似度 > 0.85，出现3次以上 | 辅助条件 |
| 主动求助 | 玩家点击「求助」按钮，或输入「提示」「不知道」「放弃」等关键词 | 立即触发 |
| 时间停滞 | 同一阶段停留超过15分钟，问题数量无增加 | 辅助条件 |

> 💡「无效问题」= AI回答否/无关，且该问题未帮助玩家排除任何错误方向。

### 3.2 阶段一：严格期（默认状态）

- **触发条件：** 游戏开始，或卡死计数器重置后
- **终止条件：** 连续5个无效问题，或玩家主动求助

**行为规则：**
- 只能回答「是」「否」「无关」三种，不添加任何解释
- 语气可以有氛围感和停顿感，但绝不给出方向性提示
- 禁止重复玩家的问题关键词（防止确认偏误）
- 禁止用语气词暗示玩家「接近了」

| 类型 | 示例 |
|------|------|
| ✅ 正确 | 玩家：「那个人是不是在浴室里死的？」→ AI：「……否。」 |
| ❌ 错误 | 玩家：「那个人是不是在浴室里死的？」→ AI：「否，你可以想想其他地点。」（主动给方向，违规） |
| ❌ 错误 | 玩家：「他是被人杀死的吗？」→ AI：「……否，他不是被杀的。」（重复关键词，违规） |

### 3.3 阶段二：引导期（卡死触发后）

- **触发条件：** 卡死计数器 ≥ 5，或玩家主动求助
- **上限：** 同一局游戏引导次数 ≤ 3次；第4次触发时直接给出预设关键线索
- **重置：** 每次引导后卡死计数器归零，重回阶段一

**行为规则：**
- 先正常回答「是/否/无关」，然后追加一句方向性暗示
- 暗示只能指向「思考维度」，不能指向「具体答案」
- 引导话术必须使用该题预设的 hint_templates，不能AI自由发挥
- 引导内容用金色边框在UI上区分显示

**引导话术模板库（按卡死类型）：**

| 卡死类型 | 引导话术示例 | 指向维度 |
|---------|------------|---------|
| 卡在地点 | 「……否。也许，地点本身不是关键——关键是那里发生了什么。」 | 事件维度 |
| 卡在人物身份 | 「……否。你已经非常接近了，也许可以想想他为什么这么做，而不是他是谁。」 | 动机维度 |
| 卡在死亡方式 | 「……无关。时间，也许比方式更值得追问。」 | 时间维度 |
| 卡在因果关系 | 「……否。这件事的起点，比你现在看到的要早得多。」 | 时序维度 |
| 完全没有方向 | 「……让我换一种方式说：如果你是故事里的主角，你最不愿意别人知道的是什么？」 | 视角维度 |

> 💡 引导的核心原则：给方向、不给答案。让玩家感觉「这是我自己想出来的」而不是「AI告诉我的」。

### 3.4 氛围感回答设计（贯穿全程）

| 情境 | ❌ 禁止版 | ✅ 推荐版 |
|------|---------|---------|
| 回答「是」 | 是 | 「……是的。」/「没错。」/「你触碰到了什么。」 |
| 回答「否」 | 否 | 「……否。」/「不是这样的。」/「这条路是死路。」 |
| 回答「无关」 | 无关 | 「这……与真相无关。」/「先放下这个。」 |
| 阶段二接近时 | 是 | 「……是的。你已经站在门口了。」（仅阶段二可用） |
| 玩家走偏时 | 否 | 「这个方向，离真相越来越远了。」（仅阶段二可用） |

### 3.5 防被套题机制

- **语义识别：**「他是自杀的吗」=「是他自己结束生命的吗」→ 视为同一问题，给出相同答案
- **反向套题防护：**「他不是X吧」→ AI统一以正向形式回答，不提供额外确认
- **连续接近防护：** 玩家连续3个问题非常接近核心答案时，AI语气变谨慎，不给额外情绪提示（阶段一）
- **变体计数：** 语义相似的问题共享同一个「已问过」标记，不重复给出新信息

### 3.6 难度与阈值的对应关系

| 难度 | 卡死触发阈值 | 引导上限 | 题目特征 |
|------|------------|---------|---------|
| ⭐ 入门 | 3个无效问题 | 5次 | 推理路径线性，线索明显 |
| ⭐⭐ 简单 | 4个无效问题 | 4次 | 2-3层推理，有一定干扰项 |
| ⭐⭐⭐ 普通 | 5个无效问题 | 3次 | 3-4层推理，关键线索需要推断 |
| ⭐⭐⭐⭐ 困难 | 7个无效问题 | 2次 | 4-5层推理，存在误导性信息 |
| ⭐⭐⭐⭐⭐ 地狱 | 10个无效问题 | 1次 | 多线叙事，真相高度反直觉 |

## 4. System Prompt 模板（可直接用于 DeepSeek API）

以下为完整的System Prompt，运行时动态注入 `{{ }}` 占位符内容：
【角色设定】
你是一位神秘的海龟汤游戏主持人。你掌握着故事的全部真相，
但你的职责是引导玩家自己推理出来，而不是直接告诉他们答案。
你说话简短、克制、有仪式感。你从不解释，从不多说一个字。
【游戏规则 - 严格执行】

只能回答「是」「否」「无关」，不能说其他实质性内容。
禁止重复玩家的问题关键词。
禁止主动提供方向性提示（除非系统标注已进入「引导模式」）。
如果玩家问超出故事范围的问题，回答「无关」。
如果玩家在问同一问题的变体，给出相同答案，不添加新信息。

【当前故事】
汤面（玩家可见）：{{SOUP_SURFACE}}
汤底（仅你知道）：{{SOUP_BOTTOM}}
核心谜点（判断用）：{{KEY_POINTS_LIST}}
推理路径层级：{{REASONING_PATH}}
【当前游戏阶段】
{{PHASE_INSTRUCTION}}
// 阶段一注入：
// 当前处于「严格期」。只回答是/否/无关，语气可有氛围感，绝不添加方向提示。
// 每条回答不超过 15 个字。
// 阶段二注入：
// 当前处于「引导期」。在回答之后追加一句方向性暗示。
// 暗示只能指向思考维度，不能指向具体答案。
// 本题引导话术参考：{{HINT_TEMPLATE}}
【回答风格】

好的回答：「……是的。」「……否。」「这，与真相无关。」
允许用省略号（……）增加停顿感和仪式感
玩家猜出全部真相时，说：「……你找到了。」然后等待系统触发汤底揭示
每条回答字数：阶段一 ≤ 15字，阶段二引导话术 ≤ 40字

## 5. 核心功能（MVP）

### P0 — MVP必做

| 功能 | 设计要点 |
|------|---------|
| 难度分级选择 | 开始游戏前选择⭐~⭐⭐⭐⭐⭐，影响AI引导阈值和题目池筛选；难度说明一句话描述预期体验 |
| 聊天式问答界面 | 玩家气泡右侧，AI回答左侧；AI回答用打字机效果逐字出现，省略号必须有停顿感 |
| AI分阶段节奏控制 | 前端维护gameState对象，每轮更新卡死计数，动态切换System Prompt的阶段注入内容 |
| 求助按钮 | 显示剩余求助次数；点击立即触发阶段二引导，消耗1次；次数耗尽变灰但仍可点击（给出关键线索） |
| 对话历史回顾 | 全程可滚动查看；AI的引导话术用金色边框标注，让玩家一眼识别转折点 |
| 汤底揭示页面 | 分层揭示：先显示「关键转折点」→ 再展开完整真相；配合动效，制造揭示仪式感 |
| 分享功能 | 分享内容 = 玩家推理路径摘要（非汤底）；格式：「我用X轮推理出了这道题，你能做到吗？」 |

### P1 — v1.1 跟进

| 功能 | 设计要点 |
|------|---------|
| 游戏大厅 | 卡片式布局；每张卡片显示：标题、难度星级、预计时长、类型标签（悬疑/伦理/反转） |
| 推理路径回顾 | 游戏结束后展示reasoning_log；标注哪些问题是关键突破，哪些是无效尝试 |

### P2 — 后续迭代

| 功能 | 设计要点 |
|------|---------|
| 用户登录与记录保存 | 游戏记录、解题历史、已解锁题目 |
| 排行榜 | 按题目统计最少问题数解题记录 |
| 自定义海龟汤 | 用户投稿题目，经审核后上线 |
| 多人对战模式 | 多人同时推理，看谁先猜出答案 |

## 6. 题目数据结构

每道题目必须包含以下字段，才能支持分阶段节奏控制和推理路径回顾：
```typescript
{
  id: "soup_001",
  title: "最后一支烟",
  difficulty: 3,              // 1-5，对应⭐~⭐⭐⭐⭐⭐
  estimated_minutes: 20,      // 预计推理时长，用于难度选择页展示
  tags: ["悬疑", "反转"],

  surface: "一个男人抽完最后一支烟后，笑着死去了。",
  bottom: "他是死刑犯，烟是刑前仪式，他笑是因为终于解脱。",

  key_points: [               // 核心谜点，AI判断用
    { point: "他是死刑犯",          importance: "critical" },
    { point: "最后一支烟是刑前仪式", importance: "critical" },
    { point: "他笑是因为解脱",       importance: "supporting" }
  ],

  reasoning_path: [           // 预设推理路径层级
    { layer: 1, clue: "那个地方有特殊规则",  hint: "场景维度" },
    { layer: 2, clue: "他的身份非常特殊",    hint: "身份维度" },
    { layer: 3, clue: "他为什么笑着接受",    hint: "动机维度" }
  ],

  hint_templates: [           // 该题专属引导话术（阶段二使用）
    { type: "identity", hint: "也许，他的身份比你想象的更特殊。" },
    { type: "emotion",  hint: "他为什么笑——比他怎么死更值得追问。" },
    { type: "context",  hint: "那个地方，以及那个时刻，有它的规则。" }
  ],

  key_clue: "这件事发生在一个有特殊规则的地方。", // 第4次求助直接给出

  reasoning_log: []           // 运行时填充，记录玩家推理过程
                              // 格式：[{ question, answer, player_inference, timestamp }]
}
```

## 7. 界面设计要求

| 页面/区域 | 设计要求 | 关键细节 |
|---------|---------|---------|
| 整体风格 | 神秘悬疑，深蓝/深紫色调 | 深色背景，浅色文字；禁用纯黑背景 |
| 难度选择页 | 开始游戏前的独立页面 | 5个难度卡片，每个显示：星级、一句话体验描述、预计时长 |
| 游戏页面 | 聊天界面风格，暗色调 | 汤面固定置顶；玩家气泡右蓝，AI气泡左深；显示有效线索数而非问题总数 |
| AI回答动效 | 打字机效果，有停顿感 | 「……」省略号必须逐字出现后停顿0.5秒，制造悬念 |
| 引导话术标注 | 阶段二引导用金色边框区分 | 让玩家一眼识别「这是特殊提示」 |
| 求助按钮 | 显示剩余次数，状态清晰 | 有次数时：「求助（剩余N次）」；耗尽时：「获取关键线索」 |
| 汤底揭示页 | 分层展开，有仪式感 | 先模糊显示 → 点击或等待3秒 → 逐层揭示 |
| 分享卡片 | 分享推理摘要而非答案 | 生成图片格式，包含：问题数、用时、关键突破问题 |

## 8. 技术栈

| 层级 | 选型 | 关键说明 |
|------|------|---------|
| 前端框架 | React + TypeScript + Tailwind CSS | 游戏状态用 Zustand 管理 |
| 游戏状态管理 | 前端 gameState 对象 | 每轮对话后更新 stuckCount/phase/hintsUsed，动态重建 System Prompt |

> 💡 DeepSeek API 调用建议：temperature=0.7；max_tokens=150（强制短回答）。

## 9. 游戏体验评测标准

每道新题上线前，必须通过以下评测（由3位测试者独立游玩后统计）：

| 评测维度 | 测试方法 | 通过标准 |
|---------|---------|---------|
| 推理时长合理性 | 3位测试者独立游玩，记录完成时长 | 平均时长在标注预计时长 ±30% 内 |
| 卡死率 | 统计全程引导触发次数 | 普通难度：引导触发次数 ≤ 2次 |
| 引导有效性 | 引导触发后10个问题内是否有突破 | 引导后突破率 ≥ 70% |
| 推理路径覆盖 | 验证 reasoning_path 各层级线索是否可被问出 | 每一层至少有3个不同问法能触发「是」 |
| 防套题 | 用10种变体问法测试同一核心谜点 | AI不给出超出规则的额外信息 |
| 氛围感评分 | 测试者完成后打分（1-5分） | 平均 ≥ 4.0分 |
| 恍然大悟感 | 测试者揭示真相时的情绪反应 | ≥ 2/3 的测试者表示「有意外感」 |

## 10. 红线清单

以下行为一旦出现，视为严重 bug，必须立即修复：

| 场景 | 禁止行为 | 正确处理方式 |
|------|---------|------------|
| 阶段一 | AI主动给出方向性提示（未经系统授权） | 严格只回答是/否/无关，任何多余的字都不输出 |
| 任意阶段 | AI回答中重复玩家问题的关键词 | 用代词或泛指替代，例如「那件事」「那个人」 |
| 阶段切换 | 卡死计数器引导后未归零，导致引导次数超限 | 每次引导触发后立即将 stuckCount 归零 |
| 防套题 | 同一问题的变体得到不一致的答案 | 语义相似度判断失效时，宁可回答「无关」也不给矛盾答案 |
| 内容安全 | 汤底在游戏过程中被AI以任何形式提前泄露 | System Prompt 中明确标注汤底「仅你知道」，且不在对话历史中出现 |
| 回答长度 | 阶段一回答超过15字，阶段二超过40字 | 在 API 调用参数中强制设置 max_tokens=150 |
| 难度失真 | ⭐难度的题实际需要60分钟才能解开 | 上线前必须通过3人评测，时长偏差超过30%不得上线
